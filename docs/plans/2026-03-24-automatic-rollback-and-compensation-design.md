# 自动回退与补偿策略设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 配置版本化与回滚设计
- 变更审批 / 发布流程设计
- 跨环境 promotion 设计
- promotion gate / 验证钩子 / 失败回退点设计
- 运行时热更新与多实例同步设计

在这些能力之上，下一步的问题是：

> 当发布、promotion、热更新、多实例同步等链路在中途失败时，系统如何形成统一的失败记录与补偿语义，而不是让失败静默消失。

本阶段目标是定义最小可用版的自动回退 / 补偿策略边界：

- 定义失败触发条件
- 定义统一补偿记录结构
- 明确补偿记录与已有控制面能力的关系
- 不做自动重试编排
- 不做自动回滚执行

## 2. 方案对比

### 方案 A：最小可用版（推荐）

只定义：
- 触发条件
- 显式补偿记录
- 人工可见、人工处理的恢复路径

优点：
- 与当前阶段最匹配
- 能快速建立失败闭环
- 风险最低

缺点：
- 自动化程度低
- 仍依赖人工处理

### 方案 B：增强版

在最小版基础上增加：
- 自动重试窗口
- 幂等补偿动作
- 失败升级策略

优点：
- 更接近成熟控制面

缺点：
- 当前阶段复杂度明显提升

### 方案 C：完整补偿工作流版

直接引入：
- 自动重试
- 自动回滚
- 多阶段补偿编排

优点：
- 自动化能力最强

缺点：
- 当前阶段完全过重
- 与现有简化控制面模型不匹配

### 结论

采用方案 A：**先定义自动回退触发条件与人工可见的补偿记录，不做自动重试编排。**

## 3. 触发条件与最小补偿记录

### 3.1 触发条件

首版至少覆盖以下失败触发条件：

- promotion gate 未通过
- 同步验证钩子失败
- 目标环境新 Released 版本创建失败
- 热更新失败
- 多实例同步失败

### 3.2 最小补偿记录模型

每次失败都生成一条结构化补偿记录，至少包含：

- `module`
- `tenant_id`
- `environment`
- `version`
- `failed_stage`
- `error_summary`
- `suggested_action`
- `created_at`

### 3.3 语义边界

首版补偿记录只要求：

- 人工可见
- 人工可追踪
- 后续可作为人工重试或人工回滚的输入依据

不要求：
- 自动消费
- 自动恢复

## 4. 与版本化 / promotion / 热更新 / 多实例同步衔接

### 4.1 与版本模型的关系

补偿记录不是新的配置版本。

因此：
- 不更新 active pointer
- 不参与版本主链
- 只附着于某次失败的配置动作之上

### 4.2 与 promotion 的关系

promotion 任一阶段失败时：
- 写补偿记录
- 不自动回滚
- 不继续后续步骤

### 4.3 与热更新的关系

热更新失败时：
- 写补偿记录
- 当前进程保留原状态或失败状态
- 不自动重试

### 4.4 与多实例同步的关系

多实例同步失败时：
- 写补偿记录
- 不自动补发
- 后续由人工决定是否重试事件传播或回滚配置

## 5. 风险与控制

### 风险 1：补偿记录和版本记录混淆

控制方式：
- 明确补偿记录不是版本
- 不进入 active pointer 决策链

### 风险 2：失败链路过多导致人工排查困难

控制方式：
- 强制 `failed_stage` 明确且有限枚举
- 强制 `suggested_action` 提供人工处理建议

### 风险 3：后续引入自动补偿时推翻首版设计

控制方式：
- 首版先稳定补偿记录模型
- 后续自动化能力围绕补偿记录展开，不重做模型

## 6. 成功标准

本阶段完成后应满足：

- 发布、promotion、热更新、多实例同步的失败都能留下统一补偿记录
- 补偿记录不干扰版本主链和当前生效指针
- 人工能够通过补偿记录知道失败位置、失败原因和建议处理动作
- 后续可在此基础上扩展自动重试与自动回滚能力
