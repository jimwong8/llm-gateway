# 2026-03-24 Admin UI 资产创建能力设计

## 1. 背景与目标

当前管理端已具备资产列表、编辑、软删除、版本回滚与统计视图能力，但缺少「从 UI 直接创建资产」入口。
本次目标是在后端内置管理页中新增「新建资产」弹窗，并接入 `POST /admin/assets`，形成创建→可见→可追踪的闭环。

## 2. 范围

### 2.1 In Scope
- 在资产列表页增加“新建资产”按钮
- 新增 Create Modal（字段录入 + 校验 + 提交）
- 前端接入 `POST /admin/assets`
- 成功后刷新列表并高亮新建记录
- 失败后保留输入并提示错误

### 2.2 Out of Scope
- 批量创建
- 创建后自动打开版本抽屉
- 后端接口语义变更（仅消费已有接口）

## 3. 校验策略（已确认）

采用“稳妥方案”：

- 必填字段：`tenant_id`、`source_model`、`task_type`、`title`、`summary`
- 可选字段：`tags`、`user_id`、`session_id`、`source_request_id`
- `tags` 输入采用逗号分割，复用现有解析逻辑（去空格、去空项）

该策略兼顾：
- 数据质量（title/summary 不为空，利于检索与治理）
- 兼容性（可选链路字段不阻塞创建）
- 与现有编辑流程一致性

## 4. 组件与代码组织

### 4.1 页面结构
- 文件：`internal/httpserver/adminui/index.html`
- 新增：
  - 资产视图顶部“新建资产”按钮
  - Create Modal（输入字段与保存/取消按钮）

### 4.2 样式
- 文件：`internal/httpserver/adminui/styles.css`
- 复用已有 modal 基础样式，新增 create 区块样式（不引入新的复杂布局体系）

### 4.3 前端逻辑
- 文件：`internal/httpserver/adminui/app.js`
- 新增函数：
  - `openCreateModal()`
  - `closeCreateModal()`
  - `saveCreateAsset()`
- 复用函数：
  - `apiPost()` 调用 `POST /admin/assets`
  - `parseTagsInput()` 解析 tags
  - `normalizeError()` 统一错误映射
- 新增状态：
  - `lastCreatedAssetID`：用于列表刷新后的行高亮定位

## 5. 数据流设计

1. 用户点击“新建资产”
2. `openCreateModal()` 打开弹窗，并默认带入当前筛选 tenant（若有）
3. 用户填写字段并点击保存
4. `saveCreateAsset()` 执行前端必填校验
5. 校验通过后调用 `apiPost("/admin/assets", payload)`
6. 成功：
   - 记录返回 `id` 到 `lastCreatedAssetID`
   - 关闭弹窗
   - 刷新资产列表
   - 在列表中高亮对应行（短时）
7. 失败：
   - 保留弹窗和用户输入
   - 在消息区展示错误文案

## 6. 错误处理与交互策略（已确认）

- API 错误映射复用已有逻辑：
  - 401：提示管理密钥问题
  - 404：提示资源不存在/已删除
  - 5xx：提示服务异常
- 保存按钮提交期间禁用，防止重复点击导致重复创建
- 创建失败不关闭弹窗，减少用户重填成本
- 成功高亮持续短时（约 6 秒）后移除

## 7. 测试与验收

### 7.1 前端交互验收
- 可打开/关闭创建弹窗
- 必填缺失时可见明确错误提示
- 提交期间按钮禁用，完成后恢复
- 成功后列表刷新且新记录高亮

### 7.2 接口联调验收
- 远程环境调用 `POST /admin/assets` 返回新 `id`
- 列表读回可看到新资产字段（title/summary/tags 等）

### 7.3 回归检查
- 不影响既有编辑流程
- 不影响版本抽屉、统计页、审计页

## 8. 风险与缓解

- 风险：后端字段要求与前端假设不一致
  - 缓解：严格按当前已验证字段组装 payload
- 风险：刷新列表后新记录不在当前页
  - 缓解：先实现“当前页可见即高亮”；后续可迭代“自动定位分页”
- 风险：重复提交产生重复数据
  - 缓解：保存按钮禁用 + 请求中状态锁

## 9. 结论

本设计在不修改后端语义的前提下，为 Admin UI 增加低风险、高价值的资产创建入口，且与现有编辑/删除交互模型保持一致，便于后续扩展批量能力与资产详情能力。
