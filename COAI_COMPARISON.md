# CoAI vs llm-gateway 前端功能对比 & 新增优化改进清单

> 基于对 https://github.com/coaidev/coai 的完整前后端源码分析
> 对比目标：llm-gateway（当前项目）+ 已研究的 new-api + 10.100.1.13 会话记忆系统

---

## 一、CoAI 技术栈 vs llm-gateway 技术栈

| 维度 | CoAI | llm-gateway（当前） | 建议 |
|------|------|-------------------|------|
| **后端** | Go 1.20 + Gin + MySQL + Redis | Go 1.22+ + PostgreSQL + Redis | ⚡ 参考 CoAI 的通道管理/适配器/计费设计 |
| **前端** | React 18 + Vite + TypeScript + Redux Toolkit + Radix UI + TailwindCSS | React 18 + Vite + TypeScript + 纯 CSS | ✅ 引入 Radix UI + TailwindCSS + Redux Toolkit |
| **状态管理** | Redux Toolkit (13 slices) | 无 | ✅ 引入 Redux Toolkit |
| **UI 组件** | Radix UI + shadcn/ui (40+ 组件) | 自研少量 | ✅ 引入 shadcn/ui |
| **实时通信** | WebSocket (gorilla/websocket) | 无 | ✅ 引入 WebSocket 流式聊天 |
| **流式响应** | SSE + WebSocket 双模式 | 无 | ✅ 实现 SSE 流式响应 |
| **认证** | JWT + API Key + DeepTrain SSO | 仅管理员 Token | ✅ 完整 JWT 认证体系 |
| **计费** | 订阅制 + 弹性计费 + 兑换码 + 邀请码 | 无 | ✅ 参考 CoAI 四层计费体系 |
| **模型支持** | 200+ 模型，21 种供应商适配器 | 6 种供应商 | ✅ 扩展适配器模式 |
| **负载均衡** | 优先级 + 加权随机 + fallback | 无 | ✅ 参考 CoAI 通道调度算法 |
| **缓存** | Redis 模型缓存 | 无 | ✅ 引入 Redis 缓存 |
| **联网搜索** | SearXNG 元搜索 | 无 | ✅ 可选集成 |
| **国际化** | i18next (5 种语言) | 无 | ✅ 引入 i18next |
| **桌面端** | Tauri 支持 | 无 | 🟢 可选 |

## 六、冲突说明

以下 CoAI 功能与 llm-gateway 现有架构**无冲突**，可直接借鉴：

| CoAI 功能 | 冲突情况 | 说明 |
|---------|---------|------|
| 流式聊天 | ✅ 无冲突 | 全新功能 |
| 用户认证体系 | ✅ 无冲突 | 扩展现有管理员 Token |
| 计费系统 | ✅ 无冲突 | 全新功能 |
| 模型适配器 | ✅ 无冲突 | 重构现有渠道管理 |
| 会话分享 | ✅ 无冲突 | 全新功能 |
| Mask 系统 | ✅ 无冲突 | 全新功能 |
| API Key 分发 | ✅ 无冲突 | 与现有租户密钥互补 |
| 模型缓存 | ✅ 无冲突 | 全新功能 |
| 联网搜索 | ✅ 无冲突 | 全新功能 |
| 国际化 | ✅ 无冲突 | 全新功能 |
| 邮件系统 | ✅ 无冲突 | 全新功能 |
| 广播通知 | ✅ 无冲突 | 全新功能 |
| PWA/Tauri | ✅ 无冲突 | 全新功能 |

**唯一需要注意的冲突**：
- llm-gateway 的"租户密钥 (TenantKeys)"与 CoAI 的"API Key"概念不同
  - llm-gateway：租户级别的 Provider API Key（BYOK）
  - CoAI：用户级别的 API 访问密钥
  - **建议**：两者共存，分别管理
