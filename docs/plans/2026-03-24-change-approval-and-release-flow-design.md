# 变更审批与发布流程设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 控制面配置管理增强设计
- 运行时热更新设计
- 多实例配置同步 / 消息总线边界设计
- 配置版本化与回滚设计

当配置具备版本与当前生效指针之后，下一步自然问题是：

> 配置修改如何经过可控的审批与发布流程，避免直接修改就立刻生效。

本阶段目标是先定义最小可用版审批 / 发布边界，不引入复杂多人审批，只建立：

- Draft
- Pending
- Released

三段状态流转。

## 2. 方案对比

### 方案 A：最小可用版（推荐）

只定义：
- Draft
- Pending
- Released

不做复杂多人审批。

优点：
- 模型清晰
- 最适合作为控制面治理第一层
- 容易与版本化、回滚、热更新衔接

缺点：
- 审批能力较基础
- 还不能表达复杂组织流程

### 方案 B：增强版

在最小版基础上加入：
- 审批意见
- 定时发布

优点：
- 更接近企业控制面

缺点：
- 当前阶段范围偏大

### 方案 C：复杂版

直接引入：
- 多级审批
- 并行审批
- 多角色会签

优点：
- 长期流程更强

缺点：
- 当前阶段明显过重
- 容易过早复杂化

### 结论

采用方案 A：**先定义 Draft、Pending、Released 三段流程。**

## 3. 状态模型与边界

### 3.1 三种状态

配置版本在首版审批流里只允许：

- `draft`
- `pending`
- `released`

### 3.2 状态语义

#### Draft

- 允许编辑
- 不参与热更新
- 不参与多实例同步
- 可继续修改或重新生成新草稿

#### Pending

- 表示已提交审批
- 当前版本被冻结
- 不允许直接改写内容
- 等待被批准进入发布

#### Released

- 表示已发布
- 才能成为当前生效版本指针候选
- 才会触发热更新与多实例同步逻辑

### 3.3 非目标状态

首版不引入：

- `rejected`
- `scheduled`
- `archived`
- `superseded`

避免状态机膨胀。

## 4. 审批与发布动作语义

### 4.1 Draft -> Pending

由“提交审批”动作触发。

语义：
- 将当前 Draft 冻结
- 记录提交时间与提交人
- 之后不可直接修改 Draft 内容

### 4.2 Pending -> Released

由“审批通过 / 发布”动作触发。

语义：
- 更新当前生效版本指针
- 触发热更新
- 触发多实例同步（若已启用）
- 记录审批人与发布时间

### 4.3 Released 的修改方式

Released 版本不能直接编辑。

若要继续修改：
- 必须基于 Released 派生新的 Draft
- 避免直接篡改已发布历史

### 4.4 回滚语义

回滚不直接把旧版本改回某个状态，而是：

- 创建一个新的 Released 版本
- 其内容引用或复制历史版本快照
- 再更新当前生效指针指向这个新版本

这样可以保证：
- 历史不可变
- 回滚动作可追踪
- 审计链路完整

## 5. 与现有版本化 / 热更新 / 同步能力的衔接

### 5.1 与版本化的关系

版本模型负责：
- 记录历史
- 保存差异
- 保存生效指针

审批 / 发布流程负责：
- 控制哪些版本有资格进入“生效路径”

### 5.2 与热更新的关系

只有 Released 版本会触发热更新。

Draft / Pending 不触发。

### 5.3 与多实例同步的关系

只有 Released 版本对应的变更事件会进入消息总线。

这样可以避免：
- 草稿误广播
- 待审批内容误进入运行态

## 6. 风险与控制

### 风险 1：Draft / Pending / Released 语义混淆

控制方式：
- 严格限制只有 Released 可生效
- Pending 冻结后不可直接编辑
- Draft 永不触发热更新

### 风险 2：审批动作与回滚动作混淆

控制方式：
- 回滚永远生成新的 Released 版本
- 不直接修改旧版本状态

### 风险 3：未来扩展多人审批时推翻首版模型

控制方式：
- 首版先稳定三态模型
- 后续审批人、审批意见、会签流程都作为附加字段或附加流程，不推翻主状态机

## 7. 成功标准

完成本阶段后应满足：

- 配置版本有清晰的三态生命周期
- 只有 Released 版本能进入当前生效指针
- Draft / Pending / Released 的动作语义明确
- 回滚行为可被纳入统一版本化与审计体系
- 后续多人审批可以在不推翻三态模型的前提下扩展
