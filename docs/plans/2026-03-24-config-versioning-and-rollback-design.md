# 配置版本化与回滚设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 控制面配置管理增强设计
- 运行时热更新设计
- 多实例配置同步与消息总线边界设计
- 实例发现与状态聚合设计

接下来的关键问题是：

> 配置如何形成可追踪的版本历史，并且在变更错误时安全回滚。

本阶段目标是定义一套增强版配置版本化与回滚能力边界：

- 统一配置版本模型
- 当前生效版本指针
- 差异视图
- 批量回滚
- 变更来源审计

## 2. 方案对比

### 方案 A：最小可用版

统一版本表 + 当前生效版本指针。

优点：
- 实现最简单
- 足够支撑基本版本追踪

缺点：
- 缺少差异视图与批量回滚能力
- 控制面价值不足

### 方案 B：增强版（推荐）

在方案 A 基础上，再加入：

- 版本差异视图
- 批量回滚
- 变更来源审计

优点：
- 与控制面长期方向一致
- 能为热更新、多实例同步提供明确版本边界
- 可解释性与可运维性更强

缺点：
- 设计复杂度高于最小版

### 方案 C：分散版

各模块各自维护版本化，不做统一版本模型。

优点：
- 实施分散，短期改动小

缺点：
- 控制面难以统一展示
- 后续热更新与多实例同步更难统一收敛

### 结论

采用方案 B：**统一配置版本表 + 当前生效指针 + 差异视图 + 批量回滚 + 变更来源审计。**

## 3. 统一版本模型与生效指针

### 3.1 版本事实模型

控制面引入统一配置版本事实表，按以下维度唯一标识一条配置版本：

- `module`
- `scope`
- `tenant_id`
- `version`

其中：
- `module`：如 policy / quota / observability / routing
- `scope`：首版可先支持 tenant 级，但预留扩展字段
- `version`：可采用递增版本号或时间戳 + hash

### 3.2 当前生效版本指针

单独维护“当前生效版本指针”，而不是覆盖历史版本。

这意味着：
- 历史版本永远不可变
- 当前生效版本可以前移或回退
- 热更新与多实例同步总是围绕“当前生效版本”工作

### 3.3 配置写入语义

每次配置写入：

1. 生成新版本
2. 写入版本事实表
3. 更新当前生效版本指针
4. 触发热更新或同步事件

## 4. 差异视图、批量回滚与审计

### 4.1 差异视图

首版只支持：

- 相邻版本 diff
- 当前生效版本 vs 指定版本 diff

明确不做：

- 三方 merge
- 复杂冲突解释
- 语义级 diff

### 4.2 批量回滚

批量回滚以 tenant 级为主。

首版允许：
- 一次将 policy / quota / observability 中的一组模块回滚到指定版本集

不要求：
- 跨 scope 事务回滚
- 自动冲突解决

### 4.3 回滚语义

回滚动作本身不直接“改写旧版本”，而是：

- 生成新的回滚事件或新版本
- 把当前生效版本指针指向目标版本或对应的新快照

这样可以保证：
- 版本历史可追踪
- 回滚本身也是可审计事件

### 4.4 变更来源审计

至少记录：

- 操作者
- 来源接口
- 模块
- 版本号
- 配置摘要
- 时间

与现有审计能力衔接时，建议保持统一事件模型，避免专门为版本化再建一套完全独立的审计语义。

## 5. 与现有热更新 / 多实例同步的关系

### 5.1 与运行时热更新的衔接

热更新只围绕“当前生效版本”触发。

这样可以确保：
- reload 的输入总是明确版本
- reload 状态可与版本号关联
- 控制面能知道“当前实例生效到哪个版本”

### 5.2 与多实例同步的衔接

多实例同步传播的是：

- 配置变更事件
- 版本号
- 生效指针变更

而不是传播原始内存配置对象。

## 6. 风险与控制

### 风险 1：统一版本模型过于抽象导致实现复杂

控制方式：
- 首版只做 tenant 级
- 首版只做少量模块
- 版本模型字段保持最小集

### 风险 2：批量回滚跨模块不一致

控制方式：
- 首版只允许显式指定版本集
- 不做自动依赖推导
- 回滚结果必须写审计并可人工核查

### 风险 3：差异视图难以泛化到所有模块

控制方式：
- 首版 diff 只做结构级和字段级展示
- 不做模块特定语义 diff

## 7. 成功标准

完成本阶段后应满足：

- 配置写入具有统一版本语义
- 当前生效版本可独立追踪
- 控制面可查看版本差异
- 控制面可做 tenant 级批量回滚
- 所有版本变更与回滚动作都有可审计来源
