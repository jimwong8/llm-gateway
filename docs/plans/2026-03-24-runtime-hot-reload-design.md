# 运行时热更新设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 路由策略引擎
- L1/L2/L3/L4 缓存与复用链路
- Billing & Observability
- Quota Observability
- Policy Engine 增强 / 深化 / 闭环
- 控制面配置管理增强的设计与实施计划

接下来的关键问题不再是“能否配置”，而是：**配置变更后，运行中的实例如何感知并在不重启进程的前提下生效。**

本阶段目标是在控制面配置管理之后，定义一套增强版运行时热更新（Runtime Hot Reload）边界：

- 支持进程内热更新
- 覆盖路由、策略、配额、观测配置
- 增加配置变更通知与实例级 reload 状态可观测
- 不追求跨实例强一致

## 2. 方案对比

### 方案 A：最小可用版

只支持进程内热更新，不做通知与状态可观测。

优点：
- 实现最简单
- 风险最低

缺点：
- 控制面无法知道 reload 是否成功
- 运维调试成本高

### 方案 B：增强版（推荐）

在进程内热更新基础上，增加：

- 配置变更通知机制（同进程内）
- 每类配置的 reload 状态视图

优点：
- 能直接支撑控制面写入后的反馈
- 便于后续演进到跨实例通知
- 对当前架构改动仍然可控

缺点：
- 需要额外维护一层运行时状态

### 方案 C：分布式强一致版

直接引入跨实例通知总线、配置版本与实例级 ACK。

优点：
- 长期最完整

缺点：
- 当前阶段复杂度过高
- 与现有系统阶段不匹配

### 结论

采用方案 B：**增强版运行时热更新**。

## 3. 热更新对象与边界

### 3.1 首版热更新对象

首版只覆盖四类运行时对象：

- 路由配置
- 策略配置
- 配额配置
- 观测配置

### 3.2 作用范围

热更新粒度限定为：

- **进程内内存态刷新**

明确不做：

- 跨实例强一致同步
- 外部消息总线驱动的分布式广播
- 配置版本仲裁

### 3.3 各模块接入边界

#### 路由配置

优先复用现有 [`SetGlobalPolicy()`](internal/router/router.go:62) 之类的运行时注入点。

#### 策略配置

通过 server 持有的 policy 运行时视图刷新，不要求重新初始化整个进程。

#### 配额配置

通过 limiter 的内存/运行时参数更新实现即时生效。

#### 观测配置

优先体现在 server 侧的启用开关、窗口参数、观测行为切换。

### 3.4 暂不纳入范围

首版不做：

- Provider Adapter 热更新
- 底层连接池重建
- OpenAI / Redis / Qdrant client 的动态切换

## 4. 通知与可观测性

### 4.1 配置变更通知

在最小进程内热更新基础上，增加配置变更通知机制，但首版只要求：

- **同进程内广播**

可以是轻量的内存事件分发，不要求外部消息总线。

### 4.2 reload 状态模型

对每类配置维护最近一次热更新状态，至少包括：

- `last_reload_at`
- `last_reload_status`
- `last_reload_error`

### 4.3 状态暴露

reload 状态可通过以下两种方式之一暴露：

- 进入现有 [`/admin/observability/*`](internal/httpserver/server.go:55) 体系
- 或新增轻量管理接口

首版更推荐进入现有 `/admin/observability/*` 或控制面接口返回结构，避免再分散管理面入口。

### 4.4 配置写入后的行为

配置写入完成后：

1. 立即触发一次同步 reload
2. 将结果写入 reload 状态视图
3. 如果“写入成功但 reload 失败”，则返回明确的状态信息供控制面感知

## 5. 测试与验证策略

### 5.1 本地测试

建议补齐：

- 路由配置变更后即时生效测试
- 配额参数变更后 limiter 行为变化测试
- reload 状态结构更新测试
- reload 失败状态测试

### 5.2 远程联调

至少验证：

- 控制面更新配置后，当前进程内行为立即变化
- reload 状态可被读取
- reload 失败时错误信息可见

### 5.3 范围约束

本阶段不测试：

- 多实例同步一致性
- 跨实例广播时序
- 版本回滚与冲突处理

## 6. 风险与控制

### 风险 1：热更新对象之间相互影响

控制方式：
- 各对象分别维护独立 reload 状态
- 按模块执行最小刷新，不做全局大重载

### 风险 2：配置写入成功但运行时未真正生效

控制方式：
- 写入后必须立即进行一次同步 reload
- 控制面只信任 reload 结果，不只信任写库结果

### 风险 3：后续演进到多实例时接口不兼容

控制方式：
- 首版状态结构中预留 `scope` / `instance` / `status` 等字段
- 将“本实例热更新”与“未来多实例通知”区分开来

## 7. 成功标准

完成后应满足：

- 路由、策略、配额、观测配置可以在进程内热更新
- 配置写入后能立即触发 reload
- 每类配置的最近一次 reload 状态可被读取
- 控制面可以区分“写入成功”与“生效成功”
- 整体不依赖进程重启即可完成配置生效
