# Promotion Gate、验证钩子与失败回退点设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 配置版本化与回滚设计
- 变更审批 / 发布流程设计
- 跨环境 promotion 设计
- 运行时热更新与多实例同步边界设计

在此基础上，下一步的关键问题是：

> 当一个 Released 版本要从 `dev` 推进到 `staging` 或从 `staging` 推进到 `prod` 时，如何在真正 promotion 之前加上最小可用的 gate、同步验证钩子与明确的失败回退点。

本阶段目标不是引入复杂流水线，而是定义：

- 手动 gate
- 同步验证钩子
- 显式失败回退点

并明确它们与版本、审计、热更新的衔接方式。

## 2. 方案对比

### 方案 A：最小可用版（推荐）

只定义：
- 手动 gate
- 同步验证钩子
- 显式失败回退点

不引入异步工作流。

优点：
- 与当前人工 promotion 模型最兼容
- 最容易先把风险控制点建立起来
- 不会过早引入复杂工作流引擎

缺点：
- 仍然需要人工驱动
- 扩展性有限

### 方案 B：增强版

在最小版基础上增加：
- 可插拔验证器
- 分阶段 gate

优点：
- 更接近正式流水线

缺点：
- 复杂度明显提升
- 当前阶段范围偏大

### 方案 C：工作流版

直接引入：
- 异步审批
- 多阶段验证
- 自动失败回退

优点：
- 自动化能力最强

缺点：
- 当前阶段过重
- 与现有简单控制面模型脱节

### 结论

采用方案 A：**先定义手动 gate、同步验证钩子与显式失败回退点。**

## 3. gate / 验证钩子 / 回退点模型

### 3.1 手动 gate

首版手动 gate 只定义：

- 是否允许从源环境推进到目标环境
- 是否允许当前 Released 版本进入 promotion

它不是自动审批链，而是一个显式检查点。

### 3.2 同步验证钩子

promotion 动作内串行执行验证钩子。

首版钩子最小返回：

- `pass`
- `fail`
- `error summary`

首版不做：
- 并行执行
- 异步执行
- 可重试编排

### 3.3 失败回退点

失败回退点首版不做自动回滚。

行为定义：
- 若 gate 未通过，promotion 不开始
- 若验证钩子失败，promotion 中止
- 目标环境不写新 Released 版本
- 记录失败结果与审计
- 后续由人工决定是否重试

### 3.4 成功边界

一旦目标环境的新 Released 版本写入成功，就认为本次 promotion 完成。

首版不自动撤销已经写入的目标版本。

## 4. 与版本 / 审计 / 热更新衔接

### 4.1 与版本模型的关系

gate 和验证钩子只作用于：
- 已 Released
- 准备 promotion

的动作之前。

它们不改变 Draft / Pending / Released 主状态机。

### 4.2 与审计的关系

无论 gate 失败还是验证钩子失败，都必须写统一审计事件，至少包括：

- `source_environment`
- `target_environment`
- `source_version`
- `hook_name` 或 gate 类型
- `result`
- `error_summary`
- `operator`
- `time`

### 4.3 与热更新的关系

只有：
- gate 通过
- 钩子通过
- 目标环境新 Released 版本创建成功

之后，才允许目标环境更新 active pointer，并触发热更新。

### 4.4 与多实例同步的关系

多实例同步同样只感知：
- 目标环境中新生成的 Released 版本

对于 gate 失败或 hook 失败，不广播任何“已生效”事件。

## 5. 风险与控制

### 风险 1：gate 与审批流职责混淆

控制方式：
- gate 只控制 promotion 是否能开始
- 审批流仍然只控制版本是否进入 Released

### 风险 2：验证失败后状态不明确

控制方式：
- 明确失败即中止
- 不写目标 Released 版本
- 写统一审计

### 风险 3：成功后无法撤销

控制方式：
- 首版明确不做自动失败回退
- 后续若要增强，另起“自动回退 / 补偿策略”设计阶段

## 6. 成功标准

本阶段完成后应满足：

- 已定义最小 gate 模型
- 已定义同步验证钩子最小接口语义
- 已定义显式失败回退点
- 已明确与版本化、审计、热更新、多实例同步的衔接关系
- 当前模型可为未来更复杂流水线能力提供稳定边界
