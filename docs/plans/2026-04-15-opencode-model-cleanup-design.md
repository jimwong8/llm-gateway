# OpenCode 可用大模型清理设计

## 背景

当前仓库同时存在网关运行时代码、OpenCode 相关依赖痕迹以及与模型/provider/路由相关的多处配置入口。用户目标不是新增模型能力，而是继续整理当前“可用的大模型”集合，并把已经失效、无效或不再能实际工作的模型项删除掉。

这类修改的风险不在于删配置本身，而在于：

1. 模型名可能在多个层面生效，包括默认值、provider 注册、router 候选、管理面展示。
2. 仓库存在两个服务入口：`cmd/server/main.go` 与 `cmd/server_main.go`，启动 wiring 不完全一致。
3. “有问题”如果没有客观标准，容易把暂时未启用的能力、mock 路径或 fallback 配置误删。

因此，本次整理采用“先验证，再删除”的策略，而不是直接凭名称做静态裁剪。

## 目标

在不引入新功能的前提下，完成 OpenCode 可用大模型集合的收敛：

- 明确当前仓库中哪些配置项会真正参与模型可用性判断。
- 找出明确失效的模型项或 provider 绑定项。
- 删除这些明确失效项，并同步修正默认模型、回退逻辑、展示逻辑与测试。
- 保证删除后服务仍能正常启动、路由仍能稳定决策、相关测试仍然通过。

## 非目标

本次工作不包括：

- 新增 provider 或接入新的外部模型平台。
- 对现有路由策略做功能性重设计。
- 顺手重构与本次删除无关的大块代码。
- 修改 node_modules 中的第三方依赖内容。

## “有问题”的判定标准

本次把“有问题”的模型项定义为满足以下至少一项，且经过代码路径或运行验证能够确认的项：

1. **无有效 provider 支撑**：模型配置指向的 provider 未注册、已被移除，或实际不会被构建进 registry。
2. **默认值悬空**：默认模型或 fallback 模型指向一个不会被 router/provider 层识别的模型名。
3. **路由永远不可命中**：模型名存在于候选配置中，但由于现有代码路径、provider 映射或过滤条件，实际上永远不会成为有效候选。
4. **运行验证明确失败**：相关测试、启动检查或最接近真实行为的验证表明该模型项一旦启用就会失败。
5. **展示脏数据**：管理端或公开接口仍暴露已失效模型项，且其来源可以明确定位到待清理配置。

以下情况不直接算“有问题”：

- 仅在 `MOCK_MODE=true` 下被标记为 disabled 的 provider。
- 文档中提到但运行时代码并未实际使用的旧名称。
- 需要外部凭证但当前本地环境未配置、而代码本身仍然有效的模型项。

## 方案比较

### 方案 A：直接删除可疑模型项

优点：
- 见效最快。
- 配置表面最干净。

缺点：
- 缺乏客观依据，容易误删。
- 删除后如果出现默认值悬空或回退失效，排查成本高。

### 方案 B：先筛查验证，再删除明确失效项（推荐）

优点：
- 依据最充分。
- 能同时覆盖 config、provider、router、展示层的一致性。
- 更适合“继续整理”而不是一次性重写。

缺点：
- 前置分析步骤更多。
- 需要补充或修正测试。

### 方案 C：先禁用，再二次物理删除

优点：
- 线上风险最低。
- 便于分阶段推进。

缺点：
- 中间态会残留无效配置。
- 用户当前需求明确要求“删除”，不如方案 B 直接。

最终采用 **方案 B**。

## 设计概要

### 1. 先锁定实际生效的模型来源

优先检查以下位置：

- `internal/config/config.go`：环境变量默认值与模型/provider 相关默认配置。
- `internal/providers/provider.go`：provider 接口能力与模型调用契约。
- `internal/providers/registry.go`：provider 注册、可用性、失败计数、熔断状态。
- `internal/router/router.go`：模型候选、打分、路由决策、fallback model。
- `internal/httpserver/server.go`：`/v1/models` 与管理端相关输出。
- `cmd/server/main.go`
- `cmd/server_main.go`

只有能影响以上真实执行路径的模型项，才纳入本次清理范围。

### 2. 建立“模型项 -> provider -> router -> 接口暴露”的映射

对每个待整理模型项，明确回答四个问题：

1. 它从哪里定义或注入？
2. 它由哪个 provider 承载？
3. 它是否会进入 router 候选或 fallback？
4. 它是否会通过 `/v1/models` 或管理接口对外暴露？

如果某模型项在其中任一层已经断裂，才继续进入删除评估。

### 3. 以最小改动删除失效项

删除动作遵守以下顺序：

1. 先删真实来源配置。
2. 再修复默认值和 fallback。
3. 再修复 registry/router 对应引用。
4. 最后清理接口展示和测试样例。

不做超出本次目标的顺手重构。

### 4. 双入口一致性检查

任何涉及启动 wiring、默认 provider、默认 model 或初始化依赖的修改，都必须同时检查：

- `cmd/server/main.go`
- `cmd/server_main.go`

避免出现一个入口已清理、另一个入口仍保留旧模型项的分叉状态。

## 错误处理与风险控制

### 风险 1：误把“环境缺失”当成“模型失效”

处理方式：
- 不以本机缺少 API Key 直接判定模型项失效。
- 重点依据代码路径完整性、注册关系和测试行为判断。

### 风险 2：删除后默认值悬空

处理方式：
- 每次删除模型项时，同时检查 `DEFAULT_PROVIDER`、`DEFAULT_MODEL` 与 router fallback。
- 必要时将默认值回退到当前代码中确认仍有效的稳定模型。

### 风险 3：对外接口仍暴露脏数据

处理方式：
- 检查 `/v1/models`、管理接口和前端展示来源。
- 删除配置后补充接口级测试，防止失效模型仍被返回。

### 风险 4：只修了一个启动入口

处理方式：
- 将两个入口都纳入文件清单与最终验证步骤。

## 验证策略

本次设计要求至少完成以下验证：

1. **配置/路由/provider 相关单测**
   - `go test ./internal/router`
   - `go test ./internal/providers`
   - 如有直接相关测试，补跑 `internal/httpserver`

2. **必要时的包级或全量兜底验证**
   - 在改动跨越多个核心模块时，执行 `go test ./...`

3. **运行时一致性检查**
   - 确认服务启动 wiring 不因删除项出错。
   - 确认 `/v1/models` 或管理面不再返回已删除失效项。

4. **静态诊断**
   - 对改动文件执行语言服务诊断，确保无新错误。

## 成功标准

满足以下条件才算完成：

- 明确失效的 OpenCode 可用大模型项已从真实来源中删除。
- 删除后不存在悬空默认模型、悬空 fallback、悬空 provider 引用。
- 两个 server 入口的相关 wiring 保持一致。
- 相关测试通过，且无新增诊断错误。
- 对外模型列表或管理展示中不再出现已确认失效的模型项。

## 实施产出

下一步将根据本设计生成实施计划，拆分为：

1. 识别真实模型来源与生效路径。
2. 建立失效项判定清单。
3. 逐处删除并修复引用。
4. 补充测试并完成验证。
