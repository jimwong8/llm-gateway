# 消息总线选型设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 进程内运行时热更新设计与实施计划
- 多实例配置同步 / 外部消息总线抽象边界设计与实施计划
- [`internal/runtime/bus.go`](internal/runtime/bus.go) 中的最小总线抽象与 [`InProcessBus`](internal/runtime/bus.go) 实现

下一阶段的目标不是立刻绑定具体中间件，而是先完成**具体消息总线选型边界**的梳理，为后续真正落地多实例配置同步时提供明确的技术方向。

本阶段聚焦比较：

- Redis Pub/Sub
- NATS
- Kafka

## 2. 方案对比

### 方案 A：Redis Pub/Sub

优点：
- 当前系统已依赖 Redis，用于缓存与限流，接入成本最低
- 适合快速试点
- 运维成本低，环境门槛小

缺点：
- 消息不持久
- 订阅方重启或短暂离线时容易丢事件
- 缺少天然的消费确认语义
- 不适合把配置传播当作强可靠链路

适用边界：
- 单机或低复杂度多实例试点
- 临时或过渡阶段的配置广播

### 方案 B：NATS（推荐）

优点：
- 对轻量广播和控制面消息非常贴合
- 订阅语义清晰
- 协议与部署模型轻量
- 后续可平滑演进到 JetStream 以获得持久化与更强可靠性

缺点：
- 需要新增基础设施组件
- 团队可能需要补齐运维与观测经验

适用边界：
- 作为长期的配置广播与控制面消息中枢
- 适合当前“消息量小、价值高、需要比 Redis Pub/Sub 更可靠”的场景

### 方案 C：Kafka

优点：
- 持久化、回放、消费语义最强
- 适合大规模事件流和复杂下游消费

缺点：
- 引入成本最高
- 对当前“配置变更广播”场景明显过重
- 运维复杂度高于当前阶段收益

适用边界：
- 未来若配置事件演化为高吞吐事件平台的一部分可再考虑
- 当前阶段不推荐纳入近期实现

## 3. 选型结论与适用边界

### 3.1 结论

首选 **NATS** 作为长期目标。

### 3.2 具体判断

- **NATS**：长期推荐
- **Redis Pub/Sub**：低门槛过渡方案
- **Kafka**：当前阶段明确不适合

### 3.3 原因

当前配置变更事件具有以下特点：

- 吞吐量低
- 配置价值高
- 希望订阅语义更清晰
- 希望后续能增强可靠性但不一开始就引入重型系统

这与 NATS 的能力形态高度匹配。

## 4. 抽象接口与演进策略

### 4.1 抽象接口

代码层应继续围绕 [`internal/runtime/bus.go`](internal/runtime/bus.go) 中的抽象接口推进：

- `PublishConfigChange(event)`
- `SubscribeConfigChange(handler)`

不把 Redis / NATS / Kafka 的 SDK 细节泄漏到上层逻辑中。

### 4.2 当前保留的实现

首版继续保留 [`InProcessBus`](internal/runtime/bus.go) 作为：

- 单实例运行方案
- 本地开发方案
- 多实例总线不可用时的回退方案

### 4.3 演进顺序

建议顺序：

1. 保持抽象接口稳定
2. 下一步优先增加 `NATSBus` 适配器
3. 如果需要低门槛试点，可增加 `RedisPubSubBus` 作为过渡实现
4. Kafka 不进入近期实施

### 4.4 协议稳定性原则

事件模型与实例状态字段必须保持中间件无关。

这意味着即使未来从 NATS 扩展到 JetStream，甚至替换为别的可靠总线，上层协议也不需要大改。

## 5. 风险与控制

### 风险 1：过早绑定 Redis Pub/Sub 导致后续可靠性不足

控制方式：
- Redis Pub/Sub 明确只作为过渡实现
- 不把它定义为长期默认方案

### 风险 2：引入 NATS 但团队运维经验不足

控制方式：
- 先把它定位为“控制面总线”，而非全局事件平台
- 控制使用范围，避免一开始承担过多职责

### 风险 3：抽象过度导致实现困难

控制方式：
- 保持总线接口最小
- 只围绕配置变更事件建模，不预先支持所有消息场景

## 6. 成功标准

完成本阶段后，应明确：

- 当前阶段的推荐总线是 **NATS**
- Redis Pub/Sub 仅作为低门槛过渡方案
- Kafka 暂不进入近期实施范围
- 后续总线实现必须继续复用 [`internal/runtime/bus.go`](internal/runtime/bus.go) 的抽象接口
- 未来实现顺序清晰：`InProcessBus` → `NATSBus` → 如有需要再提供 `RedisPubSubBus`
