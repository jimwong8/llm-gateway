# 实例发现与多实例状态聚合设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- 单实例控制面配置管理设计
- 运行时热更新设计
- 多实例配置同步与消息总线抽象边界设计
- 消息总线选型设计

下一阶段的关键问题是：

> 当配置同步和热更新都存在时，如何在多实例场景下知道“有哪些实例”“它们现在状态如何”“哪些实例已看到并应用了配置”。

本阶段目标是先定义最小可用版的实例发现与多实例状态聚合边界，不绑定具体服务发现组件。

## 2. 方案对比

### 方案 A：最小可用版（推荐）

先定义实例身份、心跳与状态聚合视图，不绑定 Consul / etcd / Kubernetes 等具体组件。

优点：
- 与当前系统阶段最匹配
- 可先稳定接口与状态模型
- 便于后续接入任意服务发现组件

缺点：
- 不能直接提供生产级自动发现实现

### 方案 B：增强版

在方案 A 基础上增加实例标签、角色分组与多维状态聚合。

优点：
- 更利于运维分群查看

缺点：
- 当前阶段容易把“状态视图”扩展成“资源编排系统”

### 方案 C：直接绑定版

直接按 Kubernetes / Consul 的发现模型来设计。

优点：
- 后续接入快

缺点：
- 过早绑定基础设施
- 限制系统可移植性

### 结论

采用方案 A：**先定义实例身份、心跳与状态聚合视图，不绑定具体服务发现组件。**

## 3. 实例身份、心跳与状态视图

### 3.1 实例身份

每个实例最少应具备以下身份字段：

- `instance_id`
- `started_at`
- `version`
- `address`

其中：
- `instance_id` 应稳定标识一个运行实例
- `version` 可用于区分二进制或配置版本
- `address` 用于管理端展示与诊断

### 3.2 心跳语义

首版心跳只要求表达：

- 该实例仍然存活
- 最近一次上报时间
- 最小运行状态摘要

心跳不绑定具体发现组件，不要求注册中心特性。

### 3.3 状态视图

首版状态视图的最小字段包括：

- `instance_id`
- `last_seen_at`
- `health`
- `role`
- `reload_status`

说明：
- `health` 可以复用健康检查摘要
- `role` 这里指实例角色，例如 `gateway`、`control-plane`、`worker` 等运行角色，而非 RBAC 角色
- `reload_status` 与已有热更新状态设计自然衔接

### 3.4 非目标

本阶段不做：

- leader 选举
- 分片归属
- 故障转移决策
- 调度器式实例管理

## 4. 状态聚合接口与一致性目标

### 4.1 接口形式

首版只暴露“当前快照”视图，可以采用：

- `/admin/runtime/instances`
- 或并入现有控制面接口中作为一个子视图

首版更推荐单独只读接口，避免控制面写接口职责过重。

### 4.2 聚合方式

聚合结果按 `instance_id` 去重，并展示每个实例最近一次：

- 心跳时间
- 健康状态
- reload 状态

### 4.3 一致性目标

首版一致性目标采用：

- **最终可收敛**

明确不要求：

- 所有实例同时可见
- 严格时钟同步
- 强一致的全局实例视图

### 4.4 时间语义

状态聚合应以“最近一次观测”为准，而不是尝试重建精确历史。

历史时间序列不属于首版目标。

## 5. 与现有热更新 / 总线设计的衔接

### 5.1 与运行时热更新衔接

现有运行时热更新中已经定义：

- `last_reload_at`
- `last_reload_status`
- `last_reload_error`

实例状态聚合层可以直接复用这些字段作为实例状态的一部分。

### 5.2 与消息总线衔接

消息总线负责传播配置事件，实例发现 / 状态聚合负责展示：

- 哪些实例看到了事件
- 哪些实例已经完成本地 reload

这样可形成“事件传播 + 实例可见性”的闭环。

## 6. 风险与控制

### 风险 1：实例身份不稳定导致聚合抖动

控制方式：
- 在设计中要求 `instance_id` 明确且稳定
- 不使用易变化的临时字段作为主键

### 风险 2：状态视图被误用为编排系统

控制方式：
- 明确首版只做可见性
- 不引入 leader / failover / scheduling 逻辑

### 风险 3：未来接入具体发现组件时接口重做

控制方式：
- 先稳定 `instance identity + heartbeat + current snapshot` 模型
- 不在当前阶段嵌入某家基础设施特有字段

## 7. 成功标准

本阶段完成后应满足：

- 已定义稳定的实例身份字段
- 已定义最小心跳语义
- 已定义多实例当前状态聚合视图
- 已明确与热更新状态、消息总线事件的衔接关系
- 后续可在不推翻接口的前提下接入具体服务发现组件
