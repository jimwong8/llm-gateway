# AI 大模型中转站设计文档

- 日期：2026-03-09
- 主题：企业级 LLM Gateway with Long-Term Cache
- 状态：已确认设计

## 1. 项目定位

本项目的第一阶段目标，是构建一个面向开发者、优先服务企业/团队接入场景的 AI 大模型中转站。对外提供与 OpenAI 兼容的 API 网关能力，对内屏蔽不同模型供应商之间的接口差异、鉴权差异、限流差异和计费差异。

和普通模型聚合网关不同，本项目从一开始就把“长期缓存”作为核心卖点。这里的长期缓存不是简单地缓存历史回答，而是沉淀可跨请求、跨会话、跨模型复用的知识资产，包括语义缓存、会话记忆、结构化事实片段、工具结果与标准化摘要。

产品形态上，系统需要从第一天起支持企业级部署约束：按可私有化交付来设计，采用控制面与数据面分离架构，以适配 SaaS、多租户专有实例和私有化部署三类未来交付方式。

## 2. 目标与非目标

### 2.1 MVP 目标

MVP 需要优先解决以下问题：

1. 为企业客户提供统一的 OpenAI 兼容接入层。
2. 支持多个主流模型供应商接入与路由。
3. 提供统一鉴权、配额控制、限流、日志和计费能力。
4. 提供基础精确缓存能力，降低重复请求成本。
5. 提供长期缓存雏形，包括语义缓存和会话记忆。
6. 为后续私有化部署和企业治理预留架构边界。

### 2.2 明确不做的事情

为避免首版失控，MVP 不应一开始就做以下内容：

1. 复杂 Agent 编排平台。
2. 大而全的可视化工作流系统。
3. 面向普通用户的完整聊天前台。
4. 全量自动化的跨模型智能编排。
5. 直接把语义缓存命中结果盲目作为最终回答返回。

## 3. 设计原则

### 3.1 稳定性优先

无论长期缓存多么重要，网关首先必须是一套可商用的稳定入口。兼容性、可观测性、限流熔断、错误隔离和供应商故障切换，是系统成立的前提。

### 3.2 长期缓存不是“大缓存表”

长期缓存的本质是知识资产层，而不是更大的 KV 缓存。系统应优先沉淀可迁移、可解释、可隔离、可更新的资产，而不是简单堆积历史原始回答。

### 3.3 跨模型复用复用的是中间资产

不同模型的措辞、风格、推理链和工具调用方式存在明显差异，系统不应直接跨模型复用完整输出文本，而应复用标准化摘要、结构化抽取结果、事实片段和工具结果等中间资产。

### 3.4 企业隔离默认最严格

缓存、记忆、日志、密钥和审计都必须以租户隔离为最高优先级。多租户只是部署模式，数据边界才是企业客户是否购买的根本。

## 4. 架构概览

推荐采用“双平面 + 双缓存扩展”的总体架构。

### 4.1 控制面

控制面负责：

- 租户管理
- API Key 生命周期管理
- 模型目录与可用性策略
- 路由策略配置
- 计费规则与配额策略
- 审计规则与安全策略
- 缓存策略和记忆策略配置

控制面不直接承载在线推理流量。它更像企业治理和配置中心，用于下发策略、管理元数据和查看运营状态。

### 4.2 数据面

数据面负责：

- 提供 OpenAI 兼容 API
- 执行鉴权和配额检查
- 执行模型路由
- 调用缓存与记忆系统
- 对接模型供应商并处理流式响应
- 记录调用日志、成本和观测指标
- 异步触发长期记忆提取任务

数据面应支持独立水平扩展，并能够按客户环境进行专有化部署。

## 5. 核心模块设计

### 5.1 API Gateway

对外暴露 OpenAI 兼容接口，包括但不限于：

- `/v1/chat/completions`
- `/v1/responses`
- `/v1/embeddings`
- `/v1/models`

其中，MVP 可优先支持 [`/v1/chat/completions`](docs/plans/2026-03-09-llm-gateway-design.md:83) 和 [`/v1/embeddings`](docs/plans/2026-03-09-llm-gateway-design.md:85)。接口兼容性要尽量贴近主流 SDK 使用习惯，降低企业迁移成本。

### 5.2 Provider Adapter

负责把外部供应商能力统一抽象为内部协议，屏蔽各家 API 的差异，包括：

- 请求格式差异
- 流式输出差异
- 工具调用差异
- JSON 输出能力差异
- token 统计口径差异
- 错误码与重试语义差异

MVP 建议优先支持 OpenAI、Anthropic、Google、Azure OpenAI，以及至少一个国产模型渠道。

### 5.3 Routing Engine

负责根据策略决定请求发往哪个模型供应商，考虑维度包括：

- 租户允许使用的模型列表
- 模型健康度
- 目标成本
- 延迟 SLA
- 区域与合规约束
- 是否要求工具调用或结构化输出

MVP 阶段路由策略可以先从静态规则 + 健康检查开始，不必一开始就引入复杂学习型路由。

### 5.4 Cache Orchestrator

缓存调度器是本项目的关键模块。它统一管理 L1、L2、L3、L4 四层缓存与记忆能力，不允许在各供应商适配器内自行处理缓存，以避免逻辑分散、命中不可解释和后续难以治理。

### 5.5 Memory Extractor

在请求成功完成后，异步从对话内容、系统摘要、工具调用结果和业务事件中提取长期记忆。该模块应支持规则抽取与模型抽取结合，避免完全依赖大模型带来的高成本和不稳定性。

### 5.6 Policy Engine

负责：

- API Key 鉴权
- RBAC / 组织级权限控制
- 配额与限流
- 敏感内容策略
- 数据边界与命名空间控制
- 供应商使用策略约束

### 5.7 Billing & Observability

负责统一记录：

- 请求次数
- token 用量
- 请求成本
- 缓存命中率
- 命中层级分布
- 平均时延
- 供应商错误率
- 热点租户与热点模型

该模块既服务运营，也服务企业客户的成本治理与审计。

## 6. 长期缓存体系设计

本项目的长期缓存体系建议采用四层结构。

### 6.1 L1：精确缓存

L1 用于处理完全或近乎完全一致的重复请求。

缓存 key 应综合以下因素：

- 规范化后的 prompt
- system prompt
- 重要上下文摘要
- model identifier
- sampling 参数
- 工具调用配置
- 输出格式约束
- 模型版本

L1 的目标是最大化降低高频重复调用成本，并实现低延迟返回。

### 6.2 L2：语义缓存

L2 用于处理语义近似请求。与传统答案缓存不同，L2 不建议在 MVP 中直接返回最终答案，而应返回候选答案、候选上下文或候选知识片段，再结合当前请求做二次生成。

L2 的关键点包括：

- 对用户请求、系统提示和上下文摘要生成 embedding
- 基于命名空间做相似检索
- 引入相似度阈值和置信度评分
- 对命中结果增加重判机制
- 保留命中来源与解释信息

这样可以降低误命中带来的业务风险。

### 6.3 L3：会话记忆

L3 不是缓存回答，而是把长期有效的信息抽取为结构化 memory item。适合保存的内容包括：

- 用户偏好
- 企业术语约定
- 长周期任务状态
- 已完成的工具结果
- 稳定事实片段
- 历史上下文摘要

每条记忆都应至少包含以下元数据：

- 所属租户
- 所属项目或空间
- 所属用户或会话范围
- 来源请求
- 时间戳
- 置信度
- TTL 或失效策略
- 可见性范围

### 6.4 L4：跨模型复用层

L4 是长期壁垒所在。该层不直接复用某个模型的完整输出，而是复用中间资产，例如：

- 标准化摘要
- 结构化字段抽取结果
- 已验证事实片段
- 工具调用结果
- 代码分析结果
- 文档摘要索引

L4 的意义在于：即使底层模型切换，上层仍可复用这些中间资产，从而提高整体调用效率与一致性。

## 7. 缓存安全与质量防护

长期缓存能力必须默认带有防护机制。

### 7.1 命中置信度阈值

所有 L2/L3/L4 命中都必须带置信度评分，低于阈值时只能作为增强上下文，不得直接作为最终输出。

### 7.2 可解释命中来源

上层调用方或内部观测系统应知道本次响应使用了哪一层缓存，命中对象来自何处，是否经过模型重判，以及最终是否被纳入生成上下文。

### 7.3 强命名空间隔离

缓存与记忆至少按以下维度隔离：

- tenant
- project
- user
- environment
- scenario

默认不得跨租户复用，也不应默认跨项目复用。

### 7.4 失效与更新机制

长期记忆必须支持：

- TTL
- 手动删除
- 批量失效
- 版本更新
- 来源追踪
- 冲突解决

否则缓存体系会随时间快速污染。

## 8. 核心请求数据流

推荐数据流如下：

1. 客户端请求进入 [`/v1/chat/completions`](docs/plans/2026-03-09-llm-gateway-design.md:211) 等兼容接口。
2. Policy Engine 执行鉴权、租户识别、配额检查和限流控制。
3. Cache Orchestrator 先查询 L1 精确缓存。
4. 若 L1 未命中，再查询 L2 语义缓存与 L3 会话记忆。
5. 系统将候选命中转换为增强上下文，而不是直接盲返。
6. Routing Engine 根据策略选择目标模型与供应商。
7. Provider Adapter 发起请求，支持流式转发。
8. 响应返回客户端。
9. 异步写入日志、计费、观测指标和 L1 缓存。
10. Memory Extractor 异步提取长期记忆，并写入 L3/L4 资产层。

## 9. 借鉴成功案例的产品判断

### 9.1 来自模型网关产品的启发

成功的模型网关产品并不是靠“支持很多模型”获得竞争力，而是靠以下能力：

- OpenAI API 兼容迁移成本低
- 请求转发稳定
- 供应商故障可回退
- 指标完整、定位迅速
- 成本透明、规则统一
- 企业治理能力足够强

这意味着你的产品必须先成为可信任入口，再去强化长期缓存卖点。

### 9.2 来自 RAG / 向量平台的启发

成功的记忆与检索平台不是“能查向量”就够了，而是把以下问题解决掉：

- 召回质量
- 更新时效
- 命中解释
- 元数据过滤
- 多租户隔离
- 成本与性能平衡

因此长期缓存一定不能只依赖向量数据库本身，还需要记忆抽取、质量控制和版本管理。

### 9.3 来自前沿技术趋势的判断

更前沿的方向正在从 prompt cache 演进为三层组合：

1. semantic cache
2. memory graph / structured memory
3. reusable artifacts

你的方向与这一趋势一致，关键是不要把这些概念都堆到 MVP，而要按阶段逐步落地。

## 10. 主要风险

### 10.1 语义缓存误命中

相似问题不等于相同问题，尤其在企业知识、时间敏感内容和配置问答场景中，误命中会产生高隐蔽性错误。

### 10.2 跨模型复用污染

不同模型间直接复用原始回答会造成风格、格式和事实层面的污染，因此必须限制复用粒度。

### 10.3 数据边界问题

企业客户会高度关注缓存是否跨用户、跨部门、跨项目泄露，因此隔离机制必须在架构级完成，而不是靠业务约定。

### 10.4 供应商接口差异过大

`streaming`、tool calling、JSON schema 支持、usage 统计等在不同供应商之间不一致，适配层复杂度会高于预期。

### 10.5 首版范围失控

如果第一版同时追求全兼容、全缓存、全治理、全前台，系统会很难稳定上线。因此必须分阶段交付。

## 11. MVP 路线图

### Phase 1：网关内核

优先交付以下内容：

- OpenAI 兼容接口
- 多模型适配
- 统一鉴权
- 配额与限流
- 基础日志与观测
- 成本统计
- L1 精确缓存

这是形成商业闭环和稳定入口的最小集合。

### Phase 2：长期缓存 v1

在稳定网关之上增加：

- L2 语义缓存
- L3 会话记忆
- 候选上下文增强
- 命中解释与置信度机制

本阶段默认不直接把语义命中当最终答案返回。

### Phase 3：跨模型复用

增加：

- 标准化摘要资产
- 结构化抽取资产
- 工具调用结果复用
- 已验证事实片段复用

这一阶段是长期壁垒的开始。

### Phase 4：企业化增强

进一步交付：

- 控制面 / 数据面分离部署
- BYOK
- 审计与合规能力
- 区域化部署
- 私有化交付体系

## 12. 技术选型建议

以下是偏务实的首版建议，而不是唯一答案。

### 12.1 服务端语言

首版建议优先选择 Go 或 Python。

- 如果更强调高并发、流式转发、网关稳定性，推荐 Go。
- 如果更强调模型生态、embedding、记忆处理和快速迭代，推荐 Python。

折中方案是：

- 数据面网关核心用 Go
- 记忆抽取与异步处理用 Python

如果团队规模较小，也可以首版统一用 Python，但需要重点优化并发和流式链路。

### 12.2 存储层建议

建议拆分存储角色：

- PostgreSQL：租户、配置、计费、审计、记忆元数据
- Redis：短期热缓存、限流、会话态热点数据
- 向量数据库：语义缓存和记忆检索
- 对象存储：日志归档、对话快照、离线分析原始材料

### 12.3 队列与异步任务

长期记忆提取、摘要生成、资产标准化和失效清理应走异步任务队列，避免阻塞主请求链路。

### 12.4 观测体系

首版就应接入 tracing、metrics、structured logging。对于网关类产品，没有可观测性就很难排查供应商差异问题和缓存误命中问题。

## 13. 成功标准

MVP 是否成功，可以用以下指标判断：

1. 企业客户可以用最少改造接入现有 OpenAI SDK。
2. 同一工作负载下，缓存使成本显著下降。
3. 语义缓存带来的错误率可控，不显著损害答案质量。
4. 多模型切换时，对业务侧代码影响极小。
5. 系统支持私有化部署演进，而不需要推翻重做。

## 14. 最终建议

一句话总结本项目的产品方向：

不是做一个“支持很多模型的中转站”，而是做一个“带长期知识资产层的企业级 LLM Gateway”。

一句话总结最关键的产品判断：

不要把长期缓存实现成更大的缓存表，而要把它实现成企业 AI 请求的知识资产层。只有这样，语义缓存、会话记忆和跨模型复用才会真正构成壁垒。
