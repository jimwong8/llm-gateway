# 多实例配置同步 / 外部消息总线设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成控制面配置管理增强设计与运行时热更新设计：

- 控制面已规划统一配置入口
- 运行时热更新已规划进程内 reload、通知与状态可观测

下一阶段的关键问题是：

> 当系统从单实例演进到多实例部署时，配置变更如何在多个实例之间传播，并最终收敛到一致状态。

本阶段目标不是选定某个具体消息中间件，而是先定义：

- 多实例配置同步的边界
- 外部消息总线的抽象接口
- 事件模型与状态模型
- 与当前进程内热更新方案的衔接方式

## 2. 方案对比

### 方案 A：最小可用版（推荐）

先定义多实例同步边界和外部消息总线抽象，不绑定具体中间件。

优点：
- 不过早绑定 Redis Pub/Sub、NATS、Kafka 等具体选型
- 能先把协议边界与状态模型稳定下来
- 有利于后续渐进式落地

缺点：
- 当前阶段还不能直接上线生产级多实例同步

### 方案 B：增强版

在方案 A 基础上，同时把 Redis Pub/Sub、NATS 等候选总线也纳入本轮设计比较。

优点：
- 后续技术选型更快

缺点：
- 设计篇幅变大
- 容易把“抽象设计”提前演变成“选型设计”

### 方案 C：直接绑定版

直接假定某个具体总线，例如 Redis Pub/Sub，并围绕它设计。

优点：
- 实施落地最快

缺点：
- 后续替换成本高
- 当前信息不足时风险大

### 结论

采用方案 A：**先定义多实例配置同步边界与消息总线抽象，不绑定具体中间件。**

## 3. 同步对象与边界

### 3.1 首版同步对象

首版只覆盖配置变更事件，不同步整个进程内存快照。

同步对象限定为四类：

- 路由配置
- 策略配置
- 配额配置
- 观测配置

### 3.2 同步粒度

传播的是“配置变更事件”，不是整个 runtime state。

原因：
- 配置是事实源
- runtime state 是实例本地派生状态
- 同步派生状态会增加复杂度且易失真

### 3.3 每个实例的行为

每个实例收到配置变更事件后：

1. 读取对应配置
2. 在本地执行 reload
3. 更新自己的 reload 状态视图

### 3.4 首版不保证的能力

明确不做：

- 严格顺序一致
- 所有实例同时完成 reload
- 强一致版本仲裁
- provider adapter / 连接池重建的跨实例同步

### 3.5 目标语义

首版只要求：

- 配置变更能传播
- 各实例最终会收敛
- 每个实例能暴露自身是否已收到并应用配置

## 4. 消息总线抽象与状态模型

### 4.1 总线抽象

首版先定义统一抽象接口，例如：

- `PublishConfigChange(event)`
- `SubscribeConfigChange(handler)`

目标是让控制面和实例之间先基于统一协议协作，而不是直接依赖具体中间件 SDK。

### 4.2 事件模型

建议事件最小字段包括：

- `module`
- `scope`
- `tenant_id`
- `version`
- `changed_at`
- `payload_ref` 或最小配置摘要

说明：
- `module` 用于路由到具体 reload handler
- `scope` 预留后续 project/environment 级扩展
- `version` 虽然首版不做仲裁，但先预留字段

### 4.3 实例级状态模型

每个实例维护自己的状态视图，至少包括：

- `last_seen_event_at`
- `last_reload_at`
- `last_reload_status`
- `last_reload_error`

这与当前单实例热更新中的状态模型形成自然扩展。

### 4.4 版本策略

首版不强制幂等版本仲裁，但预留 `version` 字段，为后续增强做准备。

后续若进入增强版，可以基于该字段实现：

- 重复事件去重
- 乱序事件忽略
- 多实例版本对账

## 5. 与现有热更新设计的衔接

### 5.1 单实例与多实例的关系

当前运行时热更新设计聚焦“进程内 reload”。

多实例同步应建立在它之上：

- 外部总线负责通知“有配置变更”
- 进程内 reload 负责在实例内部完成实际配置刷新

### 5.2 建议分层

建议分成两层：

1. **Control Plane / Bus 层**
   - 负责发布配置变更事件
2. **Runtime Reload 层**
   - 负责接收事件并在本地应用变更

这样可以避免把“消息传播”和“配置应用”耦在一起。

## 6. 测试与验证边界

### 6.1 本阶段应验证的内容

本阶段重点验证：

- 事件模型是否完整
- 总线抽象是否足够支撑后续实现
- 实例级状态模型是否可表达“已收到 / 已应用 / 应用失败”

### 6.2 本阶段不验证的内容

不要求验证：

- 真实消息中间件可用性
- 多实例时序一致性
- 跨网络抖动与重试
- broker 故障恢复

## 7. 风险与控制

### 风险 1：抽象过早导致后续实现落地困难

控制方式：
- 只定义最小接口与最小事件模型
- 不提前设计复杂 ACK、重试、分片语义

### 风险 2：进程内 reload 与跨实例同步边界混淆

控制方式：
- 明确“总线只负责传播，实例只负责应用”
- 不把实例运行时状态作为同步事实源

### 风险 3：后续总线选型适配成本高

控制方式：
- 保持总线接口最小且与中间件无关
- 将 `payload_ref` 与 `version` 提前预留

## 8. 成功标准

本阶段完成后应满足：

- 已明确多实例配置同步的对象与边界
- 已定义统一的消息总线抽象接口
- 已定义最小配置变更事件模型
- 已定义实例级 reload 状态模型
- 已明确与现有运行时热更新设计的衔接关系

这将为后续真正选定 Redis Pub/Sub、NATS 或其他中间件并落地多实例同步实现提供稳定边界。
