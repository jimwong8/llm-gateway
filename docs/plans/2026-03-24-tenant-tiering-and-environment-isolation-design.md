# 租户分层与环境隔离设计文档

日期：2026-03-24

## 1. 背景

当前系统已经完成：

- Policy Engine 增强 / 深化 / 闭环设计与实施计划
- 控制面配置管理设计与实施计划
- 配置版本化与回滚设计与实施计划
- 运行时热更新、多实例同步、实例状态聚合等后续控制面能力规划

接下来的关键问题是：

> 在控制面、版本化、审批流和热更新都存在的前提下，配置如何在“租户层级”和“环境边界”上保持严格隔离，避免不同租户和不同环境之间相互污染。

本阶段目标是先定义最小可用版边界：

- tenant_tier
- environment
- scope

并明确它们在配置模型、版本模型、审批流、热更新和多实例同步中的作用方式。

## 2. 方案对比

### 方案 A：最小可用版（推荐）

先定义：
- tenant_tier
- environment
- scope

作为统一边界模型，不进入跨租户共享策略。

优点：
- 足够支撑当前控制面后续能力演进
- 不会过早引入复杂继承和共享机制
- 易于和审批流、热更新、多实例同步对齐

缺点：
- 还不能解决 project 级和跨环境复用需求

### 方案 B：增强版

在方案 A 基础上再加入：
- project 级隔离
- 跨环境继承

优点：
- 能表达更复杂的企业配置层级

缺点：
- 当前阶段明显扩大范围
- 容易让控制面模型复杂化

### 方案 C：复杂版

直接引入：
- 跨租户共享策略
- 层层继承
- 多级 override

优点：
- 长期表达力最强

缺点：
- 当前阶段完全不适合
- 安全与治理风险最高

### 结论

采用方案 A：**先定义 tenant tier、environment 与 scope 的隔离边界，不进入跨租户共享策略。**

## 3. 租户层级与环境边界

### 3.1 tenant_tier

`tenant_tier` 用于表达租户的服务等级或能力档位，例如：

- `free`
- `pro`
- `enterprise`

它的职责是：
- 表达默认能力差异
- 表达治理阈值差异
- 表达可用功能上限差异

它**不直接等同于 RBAC**，也**不直接作为配置事实主键**。

### 3.2 environment

`environment` 用于显式隔离配置的生效范围，首版限定为：

- `dev`
- `staging`
- `prod`

控制面、版本化、审批流、热更新和多实例同步都必须把 `environment` 当作显式维度保留。

### 3.3 scope

`scope` 用于预留未来更细粒度配置作用域，目前首版可以只支持：

- `tenant`

但字段本身应保留，以便未来扩展到：
- `project`
- `workspace`
- `service`

## 4. 配置生效边界与非目标

### 4.1 生效边界

首版配置生效必须至少同时受以下约束：

- `tenant_id`
- `environment`

即使未来存在 `scope` 扩展，也不能绕开这两个维度。

### 4.2 tenant_tier 的作用方式

`tenant_tier` 只影响：
- 默认能力模板
- 默认阈值模板
- 默认治理策略模板

它不直接作为配置记录的事实主键。

### 4.3 非目标

首版明确不支持：

- 跨租户共享策略
- 跨环境继承
- project 级独立 override
- 多层回退解析链

这样可以保证当前控制面模型的安全性和可解释性。

## 5. 与已有后续能力设计的衔接

### 5.1 与控制面配置管理的关系

统一配置入口必须返回和写入：
- `tenant_id`
- `environment`
- `scope`

即使当前 scope 只有 tenant 级，也要提前保留字段。

### 5.2 与配置版本化 / 回滚的关系

版本模型至少应扩展到：
- `module`
- `tenant_id`
- `environment`
- `scope`
- `version`

这样才能保证：
- 同 tenant 不同 environment 的版本历史不相互污染
- rollback 不会误作用到错误环境

### 5.3 与审批流的关系

Draft / Pending / Released 三态流转必须同样受：
- `tenant_id`
- `environment`
- `scope`

约束。

也就是说，一个 `prod` 配置的 Pending/Released 状态，绝不能误投射到 `staging`。

### 5.4 与热更新 / 多实例同步的关系

热更新和配置同步都必须围绕：
- `tenant_id`
- `environment`
- `scope`
- `version`

进行传播与应用。

否则实例无法判断某个配置变更是否属于自己应该应用的边界。

## 6. 风险与控制

### 风险 1：dev/staging/prod 混用

控制方式：
- `environment` 成为配置、生效、回滚、同步的强制维度
- 不允许环境为空时直接进入发布链路

### 风险 2：tenant_tier 被误用成权限系统

控制方式：
- 明确 tenant_tier 只表达能力与治理档位
- 权限仍由 RBAC / Policy Engine 负责

### 风险 3：未来 project 级扩展时推翻现有模型

控制方式：
- 提前保留 `scope`
- 先让接口和事件模型支持字段，但当前只启用 tenant 级

## 7. 成功标准

本阶段完成后应满足：

- 已定义清晰的 `tenant_tier + environment + scope` 边界模型
- 已明确 `tenant_id + environment` 是最小配置生效边界
- 已明确 tenant_tier 不等于权限模型
- 已明确与控制面、版本化、审批流、热更新和多实例同步的衔接方式
- 当前模型可安全支撑后续扩展，而不需要推翻重做
